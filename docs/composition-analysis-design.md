## Vision

### 10x Check
Today's plan ("rework the page to be intuitive + persistent") is a 1x improvement over the 3,579-line monolith. The 10x version is a **Composition Balance Studio** — not a viewer of past games, but the place where you *design and tune* the game itself.

> Imagine sitting down after a play session. You open one tab. A dropdown shows last night's game. You see all four players' VP curves overlaid on their tagged compositions' ideals — instantly you spot that "Aggro Rush" overshot by 8 VP at round 12 and underdelivered by 5 by round 20, while "Slow Engine" tracked perfectly. You click "Aggro Rush." You drag two control points on the ideal curve — the saved-state indicator pulses green. You tag yesterday's "Mystery Build" by glancing at the round-by-round hex board snapshots. Done. Twenty seconds. Five compositions reviewed.

### Platonic Ideal
A new engineer joining in 6 months reads the codebase and says: *"`compositions` is the entity. `player_composition_tags` is the FK join. The three sub-routes do exactly one job each. The sidebar is shared. The chart is a component. The state is in composables. The migrations capture every schema change. The tests cover the critical state machines. Got it."*

The user feels: **calm** (one sidebar, one main area, no modal stacks), **confident** (every save is visible), **curious** (data invites exploration), **in control** (revert/export/re-tag), **productive** (≤5s to tag, ≤30s to edit a curve).

## Approach
**C: Data-Model First Rebuild** — promote `composition` to a first-class entity, then build a 3-tab UI on top.

## Accepted Scope

### A. Schema migrations (in order)
1. `0000_baseline.sql` — snapshot existing `arc-spirits-game-history` schema (9.6-A)
2. `0001_create_compositions.sql` — new `compositions` table with all 6 indexes (7.4-A):
   - `compositions(name)` UNIQUE
   - `compositions(category)`
   - `composition_tag_targets(composition_id)` FK
   - `player_composition_tags(composition_id)` FK
   - `player_composition_tags(game_id, player_color)` (verify exists)
   - `game_state_snapshots(game_id, player_color, navigation_count)` (verify exists)
3. `0002_backfill_compositions.sql` — INSERT from `composition_tag_targets`, special-case `__reference__`
4. `0003_add_composition_id_columns.sql` — add NULLABLE `composition_id` to both tag tables
5. `0004_populate_composition_id.sql` — UPDATE both tables joining on name
6. `0005_add_not_null_constraints.sql` — after dual-write soak (separate deploy)
7. `0006_drop_legacy_tag_column.sql` — third deploy, after dual-write soak

### E. New API routes

- `GET/POST/DELETE /api/player-compositions` (FK-aware, admin-gated)
- Keep `/api/composition-targets` working through dual-write phase, deprecate after Deploy 3
- `/admin/composition-analysis/health` (drift/orphan/missing counts)

### F. Tests
1. Unit: `persistedState`, `visibility.ts`, `chartConfig.ts`, `diagnosticReport.ts`
2. Integration: `compositionsService` CRUD, `tagsService` CRUD with FK
3. Migration test harness in `/supabase/migrations/__tests__/` — idempotency + invariants
4. Playwright E2E: tag-flow critical path (open game → scrub rounds → pick comp → see in library → diagnostic)

### G. Drag-to-edit ideal curves
- chartjs-plugin-dragdata for point dragging
- y-axis clamp [0, 40]
- 300ms debounce on mouse-up
- Optimistic update with rollback on save failure
- `chart.update('none')` during drag for 60fps
- "saving…" → green pulse indicator with "Last saved: HH:MM:SS" tooltip

### H. Per-game diagnostic report
- Client-side computation from already-loaded snapshots + curves
- Markdown format: per-player tracked delta, peak deviation round, integrated VP delta
- Copy-to-clipboard button
- Players with no ideal curve are skipped + noted in the report

### I. Interaction states (per screen)
Each screen designs all 4 states (Loading skeleton matching final layout, Empty CTA, Error toast w/ retry, Partial graceful labeling), all using Lantern Light brand tokens, no generic Tailwind grays.

### J. Auth
All composition-analysis routes move under `/admin/` to inherit the existing `+layout.server.ts` admin guard. New `/api/*` routes call a shared `requireAdmin()` helper.

### K. Audit / observability
- console.log every mutation server-side: `{evt, id, name, by, ts, patch}`
- `/admin/compositions/health` route for migration drift detection
- Last-saved tooltip on curve indicator
